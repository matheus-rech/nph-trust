# NPH-Trust App 1 — Project Instructions

## Overview
Patient pathway registry, dashboard, provenance engine, and attestation system for iNPH clinical research. Built with Next.js 14 + TypeScript + Tailwind + PostgreSQL (Prisma).

## Architecture
- **App 1** (this app): Pathway registry, dashboard, provenance, attestation, workflow
- **App 2** (future, shared DB): Manuscript/report/protocol generation
- Schema designed for App 2 sharing

## Authentication & Roles
- Email/password via NextAuth
- Roles: ADMIN, RESEARCHER, COORDINATOR, AUDITOR
- Demo users: admin@nphtrust.demo, researcher@nphtrust.demo, coordinator@nphtrust.demo, auditor@nphtrust.demo

## Attestation Engine (Phase 1 Refinement v2)
- **Lifecycle**: DRAFT → HASHED → SIGNED → ANCHOR_PENDING → ANCHORED → REVERIFIED (+ FAILED with retry paths)
- **Algorithm versioning**: HMAC_SHA256_v1 (current)
- **HMAC_SECRET**: Required env var, NO fallback default
- **Idempotency**: Deterministic key from projectId + eventType + payloadHash
- **Payload**: No timestamps in hashed payload (deterministic). Timestamps live on the record.
- **Separation**: Attestation engine is chain-agnostic. Blockchain logic isolated behind BlockchainProvider interface.
- **Strong binding**: `sourceArtifactIds String[]` on Attestation model — explicit references to what was attested
- **Service layer**: `lib/attestation-service.ts` — central orchestrator for create/transition/anchor/verify, wires to BlockchainProvider

## Blockchain Abstraction (Phase 1 Ready)
- `BlockchainProvider` interface: submitAnchor, verifyAnchor, getStatus, isAvailable
- `NullProvider` for graceful degradation when blockchain unavailable
- `BaseProvider` stub for Phase 1 implementation
- Provider registry with configuration-driven selection
- System functions fully without blockchain
- **AttestationService.anchorAttestation()** wires lifecycle transitions to provider calls

## Provenance Graph (Phase 1 Refinement v2)
- `ProvenanceNode` model: INPUT, TRANSFORM, OUTPUT, ATTESTATION, **EVENT**
- `ProvenanceEdge` model: directed edges with typed constants (DERIVED_FROM, ATTESTED_BY, PRODUCED, TRANSFORMED_TO, TRIGGERED_BY, RESULTED_IN)
- Full traceability from input → transform → output → attestation → event
- **RunLog linking**: ProvenanceNode.runLogId links to execution history
- **BFS lineage**: getEntityLineage() with maxDepth for scoped graph traversal
- API: recordProvenanceChain, recordEventNode, getProvenanceGraph, getEntityLineage

## Lifecycle Enforcement (Phase 1 Refinement v2)
- `lib/lifecycle.ts`: enforceAttestationTransition, enforcePathwayEventTransition
- Throws typed errors (AttestationLifecycleError, PathwayEventLifecycleError)
- Pathway events: PENDING → IN_PROGRESS → COMPLETED (terminal), with SKIPPED/CANCELLED/FAILED retry paths
- **API-level enforcement**: Events route returns 422 LIFECYCLE_VIOLATION for invalid transitions
- **PathwayEventService**: `lib/pathway-service.ts` — enforces transitions, creates RunLog, auto-attests on COMPLETED, records EVENT provenance nodes

## Pipeline Integration (Phase 2)
Every pipeline step now produces attestation + provenance node:
- **Input ingestion**: INPUT provenance node on artifact upload; attestation on import execute
- **Canonical transformation**: TRANSFORM provenance node during import execute
- **Pathway event creation**: EVENT provenance node + auto-attest on COMPLETED
- **Pathway event transitions**: EVENT provenance node for all state changes + auto-attest on COMPLETED
- **Approval/lock events**: EVENT provenance node + attestation on APPROVED/REJECTED
- **Checkpoint/version creation**: OUTPUT provenance node + attestation with snapshot hash; edges link to parent checkpoint
- **Output export (CSV/JSON)**: OUTPUT provenance node + attestation with output hash (idempotent per content)

## Phase 2 Refinement (v3)
- **Pipeline executor**: Timeout enforcement (default 30s/step), retry logic with exponential backoff (default 2 retries), structured step outcomes with timing
- **Lineage API**: `GET /api/projects/[id]/provenance/lineage?entityType=...&entityId=...&maxDepth=10` — BFS ancestor traversal
- **Traceability API**: `GET /api/projects/[id]/provenance/trace` — coverage metrics, orphan detection, node/edge distributions
- **Provenance Inspector UI**: Lineage trace view (click node → highlight ancestor chain), Traceability tab (coverage dashboard with attestation/checkpoint detail tables)
- **Partial regeneration**: COMPLETED events are terminal (no re-transition). Export tracks lockedEventCount/totalEventCount. Lifecycle violations return structured 422 with details.
- **Error handling**: Events route returns detailed LIFECYCLE_VIOLATION with currentStatus, requestedStatus, isTerminal. Warns on data modifications to COMPLETED events.
- **Integration test**: `scripts/integration-test.ts` — 11-section suite covering context, attestation, pipeline, checkpoint, graph connectivity, lineage, idempotency, error handling, degraded detection, traceability audit downgrade, duplicate attestation paths

## Phase 2 Final Hardening (v4 — FINAL)
- **Three-tier severity system**: `PipelineErrorSeverity = 'fatal' | 'degraded' | 'optional'`
  - `fatal` → pipeline FAILED, rollback triggered (data creation failures)
  - `degraded` → pipeline PARTIAL, continues but traceability gap recorded (provenance/attestation failures)
  - `optional` → warning only, pipeline stays SUCCESS (blockchain anchoring)
  - Default: `isRetryable ? 'optional' : 'fatal'`
- **Traceability audit**: Post-execution check in executor. If pipeline requires provenance/attestation but produced none → downgrade SUCCESS to PARTIAL
  - `PROVENANCE_REQUIRED_PIPELINES = ['import_execute', 'pathway_event', 'approval', 'checkpoint', 'export']`
  - `ATTESTATION_REQUIRED_PIPELINES = ['import_execute', 'checkpoint', 'export']`
- **TraceabilityReport**: Every PipelineResult now includes `traceability: { hasProvenance, hasAttestation, degradedSteps, wasDowngraded }`
- **Step-level enforcement**: All 5 Record*Provenance steps + ConditionalAttestationStep now throw `PipelineError({ severity: 'degraded' })` when context is missing (instead of silent return). Duplicate attestation paths still silently skip (correct: provenance already exists).
- **Resumability/compensation docs**: Full JSDoc on PipelineError.severity documenting retry, resume, and compensation semantics for each tier
- **RunLog**: Now records stepsDegraded count and full traceability object in outputSummary
